## Arrow confusion 
- only for cramming purposes

| scheme       | flow          |
| :----------- | :------------ |
| `lhs >> rhs` | `lhs --> rhs` |
| `lhs << rhs` | `rhs --> lhs` |

flow of the data is towards the pointing side

```bash

read-file

```

now u suddenly understand bash too

```c++

```

all (data sources) present on the **left** (`cout` && `cin` ) side.
### Concluding

| :----- | :------------------------- |

| `<<`   | stream receives the data   |

```c++

		/* two ways to interpret the line 7

	*/

	}

	if(!file.is_open()){
		cout<<"NOt aot able to open the file"<<endl;
```

## Seek and Tell

| Function  | Used for         | Meaning                    |
| --------- | ---------------- | -------------------------- |
| `seekg()` | Input (reading)  | Move the read pointer      |
| `tellg()` | Input (reading)  | Get current read position  |
| `seekp()` | Output (writing) | Move the write pointer     |
| `tellp()` | Output (writing) | Get current write position |
|           |                  |                            |
The suffixes mean:
	g = get (read)
	p = put (write)

`tell` :- tell me the position
`seek`:- go to a position

`seekg` -> move/set input (get) pointer
`seekp` -> move/set output (put) pointer

`tellg` -> get current position of input (get) pointer
`tellp` -> get current position of output (put) pointer

## Stream States
- status of the stream
- don't want to read just read the `eof()`

### `good()`

```c++
if (file.good())
{
    std::cout << "All good\n";
}
```

### `eof()`

- returns true if you go past the end

```c++
file.eof();
```
### `fail()`

```C++
int x;
/*
some file content
*/

file >> x; // finle had some string rather that the int

file.fail() //-> true

```

### `bad()`
- Serious low-level I/O error.

1. disk failure
2. hardware problem
3. file corruption
4. OS-level error

### Bit advanced section (why better)
```c++
while (!file.eof())
{
    ...
}
```

```c++
while (file >> x)
{
    std::cout << x << '\n';
}
file.clear();
```

- **Second** one is better why :- `file>>x` returns the stream, which will automatically handle the eof,invalid input, failures.

- Why we need to use **clear** :- Once you use the above code will reach the EOF and further reading will lead to errors so we have to use clear.
	- what does this clear clears?
		- all the flags we discussed in the section `fail`, `eof`,` bad`


| State    | Meaning                     |
| -------- | --------------------------- |
| `good()` | Everything okay             |
| `eof()`  | Tried reading past end      |
| `fail()` | Read/write operation failed |
| `bad()`  | Serious system-level error  |
